# ADR-0002: Provenanced ProtocolSpec

**Status:** Accepted — implementation in progress, need to verify completion of steps that follow this design choice
**Date:** 2026-05-01

## Context

In stage 2, the verified instruction is input into an LLM call, which is asked to reason and follow strict rules. For this purpose, we give the LLM information about how to extract — using examples, and also by inputting the JSON version of the definition of the Pydantic-based class ProtocolSpec.

(a) The ProtocolSpec is made mostly from steps, which are made from ExtractedSteps, which depending on instruction type contain volumes, durations, mix instructions. This enforces syntactic correctness of the protocol — that the instructions are grammatical — but only this level of validation of the LLM result would not give us any insight into whether the protocol makes sense, and whether it makes sense with respect to the instruction issued.

(b) To address the shortcoming of only enforcing syntactic correctness on the inference result of the LLM, we define 2 provenance-purposed Pydantic classes, which become types for fields of the quantity and step classes of the ProtocolSpec. Provenance is the source or origin of any object, and for our purposes produces a trace for an LLM-produced instance of our Pydantic object — whether extracted step, volume, or source within it. However, it must be noted that these provenances are generated by the LLM, based on rules given to it on how to generate these provenances. So correctness of provenance goes as far as the correctness of the model's execution.

(c) But since we now at least have claims for each atomic inferred step in the ProtocolSpec, the sanity of each single part of the specification can now be assessed. This is our best option short of deterministic interpretability outputs produced by an LLM. And since these claims are made citing at least one of `instruction`, `config`, `domain_default`, or `reasoning`, we can check each of these — very easily for the first 2, and for the latter, check the reasoning itself.

## Decision

Design the LLM to produce a provenance-first ProtocolSpec, so that from the ground up the objects that make the ProtocolSpec have a claim that can be independently verified, based on its provenance type.

## Consequences

**Easier:**
- Now we can check claims made against the instruction.

**Harder:**
- Can the LLM reliably follow these rules?
- How does one handle reasoned or domain-default type provenances? Is it LLMs all the way down?

**Now committed to:**
Now that we have a provenanced ProtocolSpec, we must design for letting the LLM fail loudly when it cannot produce an atomic step from provenance, rather than hallucinate a step. We must also verify all the blocks of the ProtocolSpec — by seeing that their provenances are true — before it gets to CompleteProtocolSpec, and only then letting it pass.

## Open questions

- Does `config` still exist as a provenance type? It should not, since we don't pass it.
- Are the examples we include in the instruction good for how we want our citations?
